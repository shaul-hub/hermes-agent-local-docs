# Domain State

## 현재 가정

이 문서는 이전 대화/작업 산출물을 현재 컨테이너에서 직접 확인할 수 없는 상태에서, 사용자의 요청에 맞춰 이어서 작업 가능한 초기 문서 형태로 재구성했습니다. 실제 기존 조사 내용이 있으면 이 문서에 병합해야 합니다.

## 도메인 구성

### 1. Resource Catalog

- 리소스 제목, 설명, 유형, 태그, 소유자, 상태를 관리합니다.
- 사용자는 업무 목적과 조건에 맞는 리소스를 검색합니다.

### 2. Governance

- 도구 승인 상태와 데이터 취급 기준을 관리합니다.
- 정책 문서와 리소스 상세 페이지를 연결합니다.

### 3. Contribution Workflow

- 구성원이 리소스를 제안합니다.
- 담당자가 검토하고 승인/수정 요청/반려합니다.
- 승인된 리소스는 카탈로그에 노출됩니다.

### 4. Learning & Enablement

- 온보딩 자료, 실습, FAQ, 사례를 관리합니다.
- 역할/팀별 학습 경로를 제공할 수 있습니다.

## 주요 엔티티 초안

| 엔티티 | 설명 |
| --- | --- |
| Resource | 플랫폼의 기본 관리 단위 |
| Tool | AI SaaS, 모델, 플러그인 등 사용 가능한 도구 |
| PromptTemplate | 재사용 가능한 프롬프트 |
| Workflow | 업무 절차와 AI 사용 단계를 묶은 템플릿 |
| Policy | 사용 규칙과 제한 사항 |
| Tag | 검색/필터링 기준 |
| Owner | 리소스 책임자 |
| Review | 리소스 품질/안전성 검토 기록 |

## 열린 질문

1. 리소스 검토 권한은 중앙팀이 갖는가, 팀별 owner가 갖는가?
2. 승인 상태는 `approved / restricted / prohibited / experimental` 정도로 충분한가?
3. 사내 SSO/권한 시스템과 어떤 수준으로 연동해야 하는가?
4. 정책 위반 리소스 신고/감사 흐름이 필요한가?
5. 리소스 품질 평점 또는 사용 피드백을 MVP에 포함할 것인가?
